     871             : Appendix:  Understanding Affix Code
     872             : 
     873             : 
     874             : An affix is either a  prefix or a suffix attached to root words to make 
     875             : other words.
     876             : 
     877             : Basically a Prefix or a Suffix is set of AffEntry objects
     878             : which store information about the prefix or suffix along 
     879             : with supporting routines to check if a word has a particular 
     880             : prefix or suffix or a combination.
     881             : 
     882             : The structure affentry is defined as follows:
     883             : 
     884             : struct affentry
     885             : {
     886             :    unsigned short aflag;    // ID used to represent the affix
     887             :    std::string strip;       // string to strip before adding affix
     888             :    std::string appnd;       // the affix string to add
     889             :    char numconds;           // the number of conditions that must be met
     890             :    char opts;               // flag: aeXPRODUCT- combine both prefix and suffix 
     891             :    char   conds[SETSIZE];   // array which encodes the conditions to be met
     892             : };
     893             : 
     894             : 
     895             : Here is a suffix borrowed from the en_US.aff file.  This file 
     896             : is whitespace delimited.
     897             : 
     898             : SFX D Y 4 
     899             : SFX D   0     e          d
     900             : SFX D   y     ied        [^aeiou]y
     901             : SFX D   0     ed         [^ey]
     902             : SFX D   0     ed         [aeiou]y
     903             : 
     904             : This information can be interpreted as follows:
     905             : 
     906             : In the first line has 4 fields
     907             : 
     908             : Field
     909             : -----
     910             : 1     SFX - indicates this is a suffix
     911             : 2     D   - is the name of the character flag which represents this suffix
     912             : 3     Y   - indicates it can be combined with prefixes (cross product)
     913             : 4     4   - indicates that sequence of 4 affentry structures are needed to
     914             :                properly store the affix information
     915             : 
     916             : The remaining lines describe the unique information for the 4 SfxEntry 
     917             : objects that make up this affix.  Each line can be interpreted
     918             : as follows: (note fields 1 and 2 are as a check against line 1 info)
     919             : 
     920             : Field
     921             : -----
     922             : 1     SFX         - indicates this is a suffix
     923             : 2     D           - is the name of the character flag for this affix
     924             : 3     y           - the string of chars to strip off before adding affix
     925             :                          (a 0 here indicates the NULL string)
     926             : 4     ied         - the string of affix characters to add
     927             : 5     [^aeiou]y   - the conditions which must be met before the affix
     928             :                     can be applied
     929             : 
     930             : Field 5 is interesting.  Since this is a suffix, field 5 tells us that
     931             : there are 2 conditions that must be met.  The first condition is that 
     932             : the next to the last character in the word must *NOT* be any of the 
     933             : following "a", "e", "i", "o" or "u".  The second condition is that
     934             : the last character of the word must end in "y".
     935             : 
     936             : So how can we encode this information concisely and be able to 
     937             : test for both conditions in a fast manner?  The answer is found
     938             : but studying the wonderful ispell code of Geoff Kuenning, et.al. 
     939             : (now available under a normal BSD license).
     940             : 
     941             : If we set up a conds array of 256 bytes indexed (0 to 255) and access it
     942             : using a character (cast to an unsigned char) of a string, we have 8 bits
     943             : of information we can store about that character.  Specifically we
     944             : could use each bit to say if that character is allowed in any of the 
     945             : last (or first for prefixes) 8 characters of the word.
     946             : 
     947             : Basically, each character at one end of the word (up to the number 
     948             : of conditions) is used to index into the conds array and the resulting 
     949             : value found there says whether the that character is valid for a 
     950             : specific character position in the word.  
     951             : 
     952             : For prefixes, it does this by setting bit 0 if that char is valid 
     953             : in the first position, bit 1 if valid in the second position, and so on. 
     954             : 
     955             : If a bit is not set, then that char is not valid for that postion in the
     956             : word.
     957             : 
     958             : If working with suffixes bit 0 is used for the character closest 
     959             : to the front, bit 1 for the next character towards the end, ..., 
     960             : with bit numconds-1 representing the last char at the end of the string. 
     961             : 
     962             : Note: since entries in the conds[] are 8 bits, only 8 conditions 
     963             : (read that only 8 character positions) can be examined at one
     964             : end of a word (the beginning for prefixes and the end for suffixes.
     965             : 
     966             : So to make this clearer, lets encode the conds array values for the 
     967             : first two affentries for the suffix D described earlier.
     968             : 
     969             : 
     970             :   For the first affentry:    
     971             :      numconds = 1             (only examine the last character)
     972             : 
     973             :      conds['e'] =  (1 << 0)   (the word must end in an E)

     975             : 
     976             :   For the second affentry:

     981             :          
     982             : 
     983             :      conds['y'] = (1 << 1)     (the last char must be a y)
     984             :      all other bits for all other entries in the conds array are zero
