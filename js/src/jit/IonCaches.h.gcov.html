      52             : // Common structure encoding the state of a polymorphic inline cache contained
      53             : // in the code for an IonScript. IonCaches are used for polymorphic operations
      54             : // where multiple implementations may be required.
      55             : //
      56             : // Roughly speaking, the cache initially jumps to an out of line fragment
      57             : // which invokes a cache function to perform the operation. The cache function
      58             : // may generate a stub to perform the operation in certain cases (e.g. a
      59             : // particular shape for an input object) and attach the stub to existing
      60             : // stubs, forming a daisy chain of tests for how to perform the operation in
      61             : // different circumstances.
      62             : //
      63             : // Eventually, if too many stubs are generated the cache function may disable
      64             : // the cache, by generating a stub to make a call and perform the operation
      65             : // within the VM.
      66             : //
      67             : // The caches initially generate a patchable jump to an out of line call
      68             : // to the cache function. Stubs are attached by appending: when attaching a
      69             : // new stub, we patch the any failure conditions in last generated stub to
      70             : // jump to the new stub. Failure conditions in the new stub jump to the cache
      71             : // function which may generate new stubs.
      72             : //
      73             : //        Control flow               Pointers
      74             : //      =======#                 ----.     .---->
      75             : //             #                     |     |
      76             : //             #======>              \-----/

      78             : // Initial state:
      79             : //
      80             : //  JIT Code
      81             : // +--------+   .---------------.

      83             : // |========|   v +----------+  |
      84             : // |== IC ==|====>| Cache Fn |  |
      85             : // |========|     +----------+  |
      86             : // |        |<=#       #        |
      87             : // |        |  #=======#        |

      96             : // |    ...     |
      97             : // +------------+

      99             : // Attaching stubs:
     100             : //
     101             : //   Patch the jump pointed to by lastJump_ to jump to the new stub. Update
     102             : //   lastJump_ to be the new stub's failure jump. The failure jump of the new
     103             : //   stub goes to the fallback label, which is the cache function. In this
     104             : //   fashion, new stubs are _appended_ to the chain of stubs, as lastJump_
     105             : //   points to the _tail_ of the stub chain.
     106             : //
     107             : //  JIT Code
     108             : // +--------+ #=======================#
     109             : // |        | #                       v
     110             : // |========| #   +----------+     +------+
     111             : // |== IC ==|=#   | Cache Fn |<====| Stub |

     116             : //     |________                |
     117             : //             |                |
     118             : //     IC      |                |
     119             : //   Entry     |                |
     120             : // +------------+               |
     121             : // | lastJump_  |---------------/
     122             : // +------------+
     123             : // |    ...     |
     124             : // +------------+
     125             : //
     126             : // While calls may be made to the cache function and other VM functions, the
     127             : // cache may still be treated as pure during optimization passes, such that
     128             : // LICM and GVN may be performed on operations around the cache as if the
     129             : // operation cannot reenter scripted code through an Invoke() or otherwise have
     130             : // unexpected behavior. This restricts the sorts of stubs which the cache can
     131             : // generate or the behaviors which called functions can have, and if a called
     132             : // function performs a possibly impure operation then the operation will be
     133             : // marked as such and the calling script will be recompiled.
     134             : //
     135             : // Similarly, despite the presence of functions and multiple stubs generated
     136             : // for a cache, the cache itself may be marked as idempotent and become hoisted
     137             : // or coalesced by LICM or GVN. This also constrains the stubs which can be
     138             : // generated for the cache.
     139             : //
     140             : // * IonCache usage
     141             : //
     142             : // IonCache is the base structure of an inline cache, which generates code stubs
     143             : // dynamically and attaches them to an IonScript.
     144             : //
     145             : // A cache must at least provide a static update function which will usualy have
     146             : // a JSContext*, followed by the cache index. The rest of the arguments of the
     147             : // update function are usualy corresponding to the register inputs of the cache,
     148             : // as it must perform the same operation as any of the stubs that it might
     149             : // produce. The update function call is handled by the visit function of
     150             : // CodeGenerator corresponding to this IC.
     151             : //
     152             : // The CodeGenerator visit function, as opposed to other visit functions, has
     153             : // two arguments. The first one is the OutOfLineUpdateCache which stores the LIR
     154             : // instruction. The second one is the IC object.  This function would be called
     155             : // once the IC is registered with the addCache function of CodeGeneratorShared.
     156             : //
     157             : // To register a cache, you must call the addCache function as follow:
     158             : //
     159             : //     MyCodeIC cache(inputReg1, inputValueReg2, outputReg);
     160             : //     if (!addCache(lir, allocateCache(cache)))
     161             : //         return false;
     162             : //
     163             : // Once the cache is allocated with the allocateCache function, any modification
     164             : // made to the cache would be ignored.
