      18             : // This code extends the AssemblerBuffer to support the pooling of values loaded
      19             : // using program-counter relative addressing modes. This is necessary with the
      20             : // ARM instruction set because it has a fixed instruction size that can not
      21             : // encode all values as immediate arguments in instructions. Pooling the values
      22             : // allows the values to be placed in large chunks which minimizes the number of
      23             : // forced branches around them in the code. This is used for loading floating
      24             : // point constants, for loading 32 bit constants on the ARMv6, for absolute
      25             : // branch targets, and in future will be needed for large branches on the ARMv6.
      26             : //
      27             : // For simplicity of the implementation, the constant pools are always placed
      28             : // after the loads referencing them. When a new constant pool load is added to
      29             : // the assembler buffer, a corresponding pool entry is added to the current
      30             : // pending pool. The finishPool() method copies the current pending pool entries
      31             : // into the assembler buffer at the current offset and patches the pending
      32             : // constant pool load instructions.
      33             : //
      34             : // Before inserting instructions or pool entries, it is necessary to determine
      35             : // if doing so would place a pending pool entry out of reach of an instruction,
      36             : // and if so then the pool must firstly be dumped. With the allocation algorithm
      37             : // used below, the recalculation of all the distances between instructions and
      38             : // their pool entries can be avoided by noting that there will be a limiting
      39             : // instruction and pool entry pair that does not change when inserting more
      40             : // instructions. Adding more instructions makes the same increase to the
      41             : // distance, between instructions and their pool entries, for all such
      42             : // pairs. This pair is recorded as the limiter, and it is updated when new pool
      43             : // entries are added, see updateLimiter()
      44             : //
      45             : // The pools consist of: a guard instruction that branches around the pool, a
      46             : // header word that helps identify a pool in the instruction stream, and then
      47             : // the pool entries allocated in units of words. The guard instruction could be
      48             : // omitted if control does not reach the pool, and this is referred to as a
      49             : // natural guard below, but for simplicity the guard branch is always
      50             : // emitted. The pool header is an identifiable word that in combination with the
      51             : // guard uniquely identifies a pool in the instruction stream. The header also
      52             : // encodes the pool size and a flag indicating if the guard is natural. It is
      53             : // possible to iterate through the code instructions skipping or examining the
      54             : // pools. E.g. it might be necessary to skip pools when search for, or patching,
      55             : // an instruction sequence.
      56             : //
      57             : // It is often required to keep a reference to a pool entry, to patch it after
      58             : // the buffer is finished. Each pool entry is assigned a unique index, counting
      59             : // up from zero (see the poolEntryCount slot below). These can be mapped back to
      60             : // the offset of the pool entry in the finished buffer, see poolEntryOffset().
      61             : //
      62             : // The code supports no-pool regions, and for these the size of the region, in
      63             : // instructions, must be supplied. This size is used to determine if inserting
      64             : // the instructions would place a pool entry out of range, and if so then a pool
      65             : // is firstly flushed. The DEBUG code checks that the emitted code is within the
      66             : // supplied size to detect programming errors. See enterNoPool() and
      67             : // leaveNoPool().
