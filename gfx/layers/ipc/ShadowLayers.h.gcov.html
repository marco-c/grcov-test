      78             : /**
      79             :  * We want to share layer trees across thread contexts and address
      80             :  * spaces for several reasons; chief among them
      81             :  *
      82             :  *  - a parent process can paint a child process's layer tree while
      83             :  *    the child process is blocked, say on content script.  This is
      84             :  *    important on mobile devices where UI responsiveness is key.
      85             :  *
      86             :  *  - a dedicated "compositor" process can asynchronously (wrt the
      87             :  *    browser process) composite and animate layer trees, allowing a
      88             :  *    form of pipeline parallelism between compositor/browser/content
      89             :  *
      90             :  *  - a dedicated "compositor" process can take all responsibility for
      91             :  *    accessing the GPU, which is desirable on systems with
      92             :  *    buggy/leaky drivers because the compositor process can die while
      93             :  *    browser and content live on (and failover mechanisms can be
      94             :  *    installed to quickly bring up a replacement compositor)
      95             :  *
      96             :  * The Layers model has a crisply defined API, which makes it easy to
      97             :  * safely "share" layer trees.  The ShadowLayers API extends Layers to
      98             :  * allow a remote, parent process to access a child process's layer
      99             :  * tree.
     100             :  *
     101             :  * ShadowLayerForwarder publishes a child context's layer tree to a
     102             :  * parent context.  This comprises recording layer-tree modifications
     103             :  * into atomic transactions and pushing them over IPC.
     104             :  *
     105             :  * LayerManagerComposite grafts layer subtrees published by child-context
     106             :  * ShadowLayerForwarder(s) into a parent-context layer tree.
     107             :  *
     108             :  * (Advanced note: because our process tree may have a height >2, a
     109             :  * non-leaf subprocess may both receive updates from child processes
     110             :  * and publish them to parent processes.  Put another way,
     111             :  * LayerManagers may be both LayerManagerComposites and
     112             :  * ShadowLayerForwarders.)
     113             :  *
     114             :  * There are only shadow types for layers that have different shadow
     115             :  * vs. not-shadow behavior.  ColorLayers and ContainerLayers behave
     116             :  * the same way in both regimes (so far).
     117             :  *
     118             :  *
     119             :  * The mecanism to shadow the layer tree on the compositor through IPC works as
     120             :  * follows:
     121             :  * The layer tree is managed on the content thread, and shadowed in the compositor
     122             :  * thread. The shadow layer tree is only kept in sync with whatever happens in
     123             :  * the content thread. To do this we use IPDL protocols. IPDL is a domain
     124             :  * specific language that describes how two processes or thread should
     125             :  * communicate. C++ code is generated from .ipdl files to implement the message
     126             :  * passing, synchronization and serialization logic. To use the generated code
     127             :  * we implement classes that inherit the generated IPDL actor. the ipdl actors
     128             :  * of a protocol PX are PXChild or PXParent (the generated class), and we
     129             :  * conventionally implement XChild and XParent. The Parent side of the protocol
     130             :  * is the one that lives on the compositor thread. Think of IPDL actors as
     131             :  * endpoints of communication. they are useful to send messages and also to
     132             :  * dispatch the message to the right actor on the other side. One nice property
     133             :  * of an IPDL actor is that when an actor, say PXChild is sent in a message, the
     134             :  * PXParent comes out in the other side. we use this property a lot to dispatch
     135             :  * messages to the right layers and compositable, each of which have their own
     136             :  * ipdl actor on both side.
     137             :  *
     138             :  * Most of the synchronization logic happens in layer transactions and
     139             :  * compositable transactions.
     140             :  * A transaction is a set of changes to the layers and/or the compositables
     141             :  * that are sent and applied together to the compositor thread to keep the
     142             :  * LayerComposite in a coherent state.
     143             :  * Layer transactions maintain the shape of the shadow layer tree, and
     144             :  * synchronize the texture data held by compositables. Layer transactions
     145             :  * are always between the content thread and the compositor thread.
     146             :  * Compositable transactions are subset of a layer transaction with which only
     147             :  * compositables and textures can be manipulated, and does not always originate
     148             :  * from the content thread. (See CompositableForwarder.h and ImageBridgeChild.h)
     149             :  */
